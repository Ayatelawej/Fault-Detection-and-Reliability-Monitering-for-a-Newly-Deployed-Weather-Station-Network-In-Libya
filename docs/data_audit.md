## Station registry source

The station registry was lifted from the companion Mozn post-processing
project, which uses the same station-hour dataset. The source registry column
`elevation_m` was renamed to `elevation` to match this project's merged dataset
schema. The enriched registry now lives at `data/merged/station_registry.csv`.

The registry enrichment is reproducible with:

```bash
python -m src.features.build_station_registry
```

## Row-state definitions

The canonical row-state artifact is generated by
`src.features.row_state.classify_row_states`. It assigns one value per
station-hour using the documented priority order below.

- `online_complete`: The station has `data_present == 1` and every configured measurement column is non-null.
- `online_partial_missing`: The station has `data_present == 1`, but at least one configured measurement column is null. These rows represent online periods with possible partial sensor dropout.
- `true_outage_candidate`: The station has `data_present == 0` between its first and last present timestamps. The availability engine treats these rows as candidate outages.
- `terminal_padded_absence`: The station has `data_present == 0` after its last present timestamp. These rows are padded end-of-record absences, not primary outage evidence.
- `warmup`: The station has `data_present == 1` within the first 7 days after its first present timestamp. These rows are kept separate because early installation behavior may be atypical.
- `before_first_present`: The row timestamp is before the station's first present timestamp. These are pre-activation padded rows when they appear.
- `station_never_active`: The station has no rows with `data_present == 1`. This state is reserved for inactive registry entries.
- `invalid_missing_id_or_time`: The row is missing `station_id` or `hour_utc`. This state has highest priority because station-hour identity is required for downstream use.

## Status_class definitions

`src.features.build_station_registry` enriches the registry with
`present_rate`, `terminal_padded_rows`, and `status_class`. Status classes use
the following first-match logic:

- `outage_dominated`: `present_rate < 0.55` or `terminal_padded_rows >= 168`.
- `weak`: `0.55 <= present_rate < 0.70`.
- `usable`: `0.70 <= present_rate < 0.85`.
- `reliable_but_terminal_gap`: `present_rate >= 0.85` and `terminal_padded_rows > 0`.
- `reliable_reference_candidate`: `present_rate >= 0.85` and `terminal_padded_rows == 0`.

## Downstream use

- Availability events use `true_outage_candidate` rows only, avoiding terminal padding and warmup periods.
- Neighbor features prefer `reliable_reference_candidate` stations as reference stations.
- Label review uses the `row_state` column as a starting point for the review queue.
